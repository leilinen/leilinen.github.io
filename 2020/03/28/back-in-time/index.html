借助savepoint使Flink任务时光倒流
2020-03-28

在持续计算的过程中，会经常遇到因为一些问题到重新消费数据的情况。我们可以通过savepoint实现任务重启到某一个时间点进行重新计算。我们设计了一个系统来实现这一功能。

流式计算一个特点就是程序会不断的进行计算，没有任务终止的时候；流式计算任务在运行过程中，会不断生成任务状态，并且把任务状态保存下来。Flink程序就是一种流式计算程序。

在Flink中，有两种状态触发方式：savepoint和checkpoint。那么他们俩有什么区别那？

checkpoint

任务在运行过程中，会根据用户的设置不断触发checkpoint，记录当前任务状态。checkpoint 可以保存在内存，HDFS, ROCKSDB中。checkpoint数据写入是异步的，因此Flink任务在做checkpoint过程中可以不影响正常数据处理。 checkpoint可以分为增量和全量两种。checkpoint 的作用是作为Flink的恢复机制，当任务出现故障时，确保可以自动恢复。

savepoint

savepoint是一种特殊的checkpoint, 记录了任务的全量数据，由用户手动触发，一般会保存在HDFS中。当任务重启时，可以指定savepoint，读取savepoint的状态，从当前savepoint开始运行。

在持续计算的过程中，会经常遇到因为一些问题到重新消费数据的情况。我们可以通过savepoint实现任务重启到某一个时间点进行重新计算。我们设计了一个系统来实现这一功能。

实现时光倒流

Flink任务时光倒流的机制，就是借助savepoint，从指定的savepoint位置重新运行。为了实现Flink任务时光倒流的目标，我们做了一个打点系统：

Flink任务在运行的时候，我们的打点程序会定期请求任务的savepoint，并把Savepoint文件存储到指定的路径下。

// 生成savepoint的命令
nohup flink savepoint {job_id} {savepoint_path} -yid {application_id}

为了满足高可用的需求，我们允许Flink任务可以提交到不同集群上。同样在切换集群的时候，我们也要支持Flink任务时光倒流。所以这时需要一个定时任务，可以将hdfs1上的savepoint文件同步到hdfs2上。实现这个功能有比较多的解决办法，我们采用的是distcp命令进行同步。

随着时间的推移，将会有越来越多的savepoint文件存储的hdfs上，这会对hdfs造成压力。而实际用户的重新启动任务的需求，会随着时间的推移而越来越少。也就是说，用户一般不会到一个比较久的时间点去重启任务。所以我们会对保存超过3天的savepoint文件进行清理。当然这也是一个定时任务在扫描清理。

总结

简单来说，让Flink任务时光倒流，就是不断的去生成savepoint文件，然后在启动任务的时候指定某个savepoint启动。
