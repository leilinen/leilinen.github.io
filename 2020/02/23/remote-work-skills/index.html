远程办公守则
2020-02-23

远程办公很麻烦？教你几个技巧。

2020年伊始，一场疫情打乱了很多人的节奏，原本年初六上班的公司也不断延期开工。好在我们是通过网络工作的，大多数有条件的公司选择了远程办公方式工作—这也是无奈的之举。

从2月3号到现在，已经做了三周的远程工作，从一开始的新奇到逐渐的适应，渐渐觉得和正常上班没有什么不同。在这三周的工作中，完成了两次上线工作，解决了无数个问题，同时也产生了新的问题。

远程办公逐渐为人们所接受，实际上在欧美一些国家，很多公司都已经开始推行远程办公的工作方式，远程办公有着很多的好处：

  • 不需要大面积的办公场所，节省房租支出；
  • 可以不受地域限制招揽人才；

但是为什么我们的公司不愿意实施远程办公那？从管理层上讲，老板们很不“放心”，很多老板担心员工在没有监视的情况下“划水”，“凭什么我支出了工资而你在玩儿？“ 还有些老板恨不得一天24小时让员工工作，没有在老板眼皮底下工作让老板非常的不安心。能否实施远程办公的工作方法，其实就是能否抹除老板和员工之间的不信任感。目前来看是挺困难的。

从员工的角度来说，能够做到远程办公的员工，首先要具有高度的责任感，能够做到“自我驱动”，主动去完成工作，而不是等待别人的催促；其次要对工作内容比较熟练，能够比较快的解决问题（工作，其实就是帮老板解决问题）；还要具有主动性，主动完成工作，主动与同事沟通。

在协同办公的过程中，信息沟通是一个很困难的问题，特别是对于一个复杂的项目，成员之间经常会出现信息壁垒，比如对于一个业务A，同事甲熟悉而同事乙一无所知，这其实是一个很糟糕的情况。远程办公情况下，信息壁垒问题会更加突出，如何跨越信息壁垒问题也是我们该思考的。以我们的经验，一般会做到以下几点：

  • 重要的问题记录在confluence下，方便同事查阅；
  • IM上，尽量在群里沟通，而不是一对一沟通；

彻底清除信息壁垒是不现实的，但是尽量做到信息透明，对于每一个成员都是很有意义的。

远程办公也是有很多技巧的，这里摘录了一些常用的工具：

  • GitHub：代码托管，公开的 RFC，社区 Issue 反馈，产品发布，Code Review 等。
  • Zoom：在线会议。
  • Slack：即时通讯，机器人消息中枢。
  • 即时通讯：钉钉，企业微信等。
  • 在线文档：文档协作，幻灯片，表格。
  • 邮件，日历。
  • Confluence：内部的文档，包括已成型的设计文档（如内部的 RFC 文档），Wiki 等。
  • Jira：Bug 和 Milestone 跟踪。
  • Trello：看板，记录一些重要客户和事件的备忘。
  • Jenkins：持续集成，daily build。

优秀工程师的创造力是无穷的，尤其在远程工作的背景下，非常鼓励工程师通过自制工具来提升工作的效率。能够交给机器做的事情，我们绝不会让人去做。

开会也是日常工作的一个重要事项，经常出现会议上主讲人长篇大论，下面的人在昏昏欲睡。我们也希望制定开会守则，内容如下：

  • 对于类似例会这种偏信息拉平的会议，Manager最好不要直接在这类会议上做决策；
  • 善用Calendar
  • 会议前发Agenda(议程)，会议后形成Meeting notes发给参会的人，并记录的wiki中
  • 尽量少开大会长会 (lianggePizza原则)

最后，希望我们每一个人都能够热爱生活，热爱工作！
